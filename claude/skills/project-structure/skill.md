# Project Structure 스킬

프로젝트를 분석하여 AI가 코드베이스를 빠르게 파악할 수 있는 구조화된 명세 문서를 생성한다.

## 핵심 원칙

1. **AI 최적화**: 사람이 읽기 좋은 문서가 아니라, AI가 코드베이스를 빠르게 이해하는 데 최적화된 문서
2. **간결함**: 코드에서 직접 파악 가능한 내용은 최소화, 구조적 맥락과 관계에 집중
3. **정확성**: 추측하지 않음. 실제 파일과 코드를 기반으로 작성
4. **유지보수성**: 프로젝트 변경 시 쉽게 갱신 가능한 구조

## 분석 순서

```
1. 프로젝트 루트에서 기술 스택 감지 파일 탐색
2. 디렉토리 트리 수집 (제외 목록 적용)
3. 엔트리포인트 → 의존성 주입 → 서비스 등록 순으로 추적
4. 컨트롤러/라우트 → 서비스 → 데이터 접근 계층 매핑
5. 설정 파일에서 외부 연동 파악
6. 템플릿에 맞춰 문서 생성
```

## 기술 스택별 분석 포인트

### .NET (ASP.NET Core)

| 분석 대상 | 확인 파일 | 추출 정보 |
|----------|----------|----------|
| 서비스 등록 | `Program.cs`, `Extensions/` | DI 등록 패턴, 미들웨어 파이프라인 |
| DB 컨텍스트 | `*DbContext.cs` | 컨텍스트명, 연결 DB, 주요 엔티티 |
| API 엔드포인트 | `Controllers/` | 라우트 패턴, HTTP 메서드, 그룹핑 |
| 인증/인가 | `Extensions/Auth*`, 미들웨어 | 인증 방식 (JWT, Cookie 등) |
| 외부 연동 | `appsettings.json`, DI 등록 | Blob, SendGrid, Redis 등 |
| 모델/DTO | `Models/`, `Dtos/` | 주요 도메인 모델 목록 |

### Vue.js / Quasar

| 분석 대상 | 확인 파일 | 추출 정보 |
|----------|----------|----------|
| 라우팅 | `router/routes.ts` | 페이지 구조, 네비게이션 가드 |
| 상태관리 | `stores/` | Store 목록, 주요 상태 |
| 컴포넌트 | `components/`, `pages/` | 컴포넌트 계층, 공유 컴포넌트 |
| API 통신 | `api/`, `services/` | API 클라이언트 패턴 |
| 레이아웃 | `layouts/` | 레이아웃 구조 |
| 플러그인 | `plugins/`, `boot/` | 초기화 로직 |

### Python (Django / FastAPI / Flask)

| 분석 대상 | 확인 파일 | 추출 정보 |
|----------|----------|----------|
| 엔트리포인트 | `manage.py`, `app.py`, `main.py` | 앱 초기화 방식 |
| 라우팅 | `urls.py`, 데코레이터 | URL 패턴 |
| 모델 | `models.py`, `models/` | ORM 모델 구조 |
| 설정 | `settings.py`, `.env` | 환경별 설정 |

## 문서 크기 제어

| 프로젝트 규모 | 디렉토리 깊이 | API 표시 | 엔티티 표시 |
|-------------|-------------|---------|-----------|
| 소규모 (<20 파일) | 전체 | 전체 목록 | 전체 목록 |
| 중규모 (20~100 파일) | 3단계 | 그룹 요약 | 주요 엔티티만 |
| 대규모 (100+ 파일) | 2단계 | 그룹 요약 | 컨텍스트별 요약 |

**최대 목표: 300줄 이내**

줄 수 초과 시 조정 순서:
1. API 엔드포인트를 그룹 요약으로 축약
2. 엔티티를 컨텍스트별 개수만 표시
3. 디렉토리 트리 깊이 축소

## 제외 사항

문서에 포함하지 않는 정보:
- 연결 문자열, API 키, 토큰 등 민감 정보
- 코드 스니펫 (파일 경로로 참조)
- 변경 이력 (git log로 확인)
- 상세 설정값 (설정 파일 직접 참조)
- node_modules, bin/obj 등 빌드 산출물
