# Analyze Orchestration

## 언제 사용하나

- 사용자가 `/analyze:orchestration`와 같은 의도로 orchestration 수행 방식을 되돌아보려 할 때
- 작업이 느렸거나, 과하게 조사했거나, 검증이 약했거나, 병렬화가 애매했다고 느낄 때
- 반복되는 작업 패턴에서 다음 번 운영 규칙을 개선하고 싶을 때

## source mapping

- Claude command: `/analyze:orchestration`
- Source file: `claude/commands/analyze/orchestration.md`

## Codex 기준 핵심 해석

- 이 스킬은 `~/.claude/orchestration/reports/*.json` 같은 고정 경로를 전제하지 않는다.
- 자동 저장된 타임라인이나 JSON 보고서가 없어도, 현재 스레드와 로컬 아티팩트만으로 회고할 수 있어야 한다.
- 기본값은 `현재 작업 또는 사용자가 지정한 로컬 파일` 분석이다.
- 영구적인 학습 규칙 파일 업데이트는 기본 동작이 아니다. 사용자가 원할 때만 문서나 스킬을 수정한다.
- local feedback hook이 켜져 있다면, 그 로그를 기본 증거 소스로 우선 사용한다.

## 분석 대상 우선순위

가능한 증거를 아래 순서로 사용한다.

1. local feedback hook 로그와 artifact
2. 현재 스레드의 작업 흐름
3. 로컬 git diff와 변경 파일
4. 테스트, 빌드, 린트, 로그 출력
5. 사용자가 지정한 로컬 report/note 파일
6. 사용자가 명시한 디렉터리 안의 여러 report 파일

증거가 부족하면 추정으로 채우지 말고 부족한 점을 밝힌다.

### hook 로그 사용 원칙

- hook 로그가 있으면 별도 orchestration report보다 먼저 본다.
- hook 로그는 실제 tool 실행, 검증 수행 여부, stop gate 동작을 보여주므로 기본 회고 증거로 적합하다.
- 별도 report 파일은 사용자가 직접 만들었거나 지정한 경우에만 보조 증거로 쓴다.
- hook 로그와 별도 report가 충돌하면, 측정 근거가 더 직접적인 쪽을 우선한다.

## 옵션 해석

사용자가 source command 방식으로 말하면 아래처럼 해석한다.

- 기본: 현재 orchestration 작업 또는 가장 가까운 로컬 증거를 분석
- `--report <path>`: 지정한 로컬 파일 하나를 우선 분석
- `--all`: 사용자가 지정한 디렉터리나 명확한 report 집합이 있을 때만 다건 분석

`--all`인데 분석할 파일 집합이 없으면, 파일이 없다고 보고하고 현재 스레드 분석으로 임의 전환하지 않는다.

## 분석 흐름

### 1단계 - 증거 수집

- 어떤 run을 분석하는지 범위를 정한다
- 관련 hook 로그, diff, 검증 결과, 작업 메모, 로컬 파일을 모은다
- 증거가 충분한지 먼저 판단한다

### 2단계 - 실행 재구성

- 이 작업이 `quick`, `standard`, `deep` 중 어떤 모드에 가까웠는지 본다
- 실제로는 어느 단계에서 시간이 많이 새었는지 정성적으로 재구성한다
- 불필요한 탐색, 늦은 계획, 과한 병렬화, 부족한 검증 여부를 본다

### 3단계 - 진단

다음 항목을 우선순위로 본다.

- 모드 선택이 적절했는가
- 조사량이 작업 크기에 비해 과했는가
- 계획이 부족해서 재작업이 생겼는가
- write scope 판단이 적절했는가
- 병렬화가 실제로 이득이었는가
- 검증이 변경 위험에 비해 충분했는가
- 멈춰야 할 지점에서 밀어붙이지 않았는가

### 4단계 - 결과 보고

결과 보고는 아래 구조를 기본으로 한다.

```md
## 요약
- 분석 대상:
- 작업 성격:
- 추정 모드:

## 핵심 판단
1. ...
2. ...

## 병목과 원인
- ...

## 검증 품질
- 수행한 검증:
- 부족했던 검증:

## 다음 번 개선 규칙
- 즉시 적용:
- 상황부 적용:
- 보류:
```

### 5단계 - 선택적 업데이트

사용자가 원할 때만 아래를 수행한다.

- 관련 skill 문서 수정
- 운영 규칙 문서 추가/갱신
- 반복 패턴을 playbook 형태로 정리

기본값은 `분석만 하고 종료`다.

## 판단 기준

### 좋은 orchestration 신호

- 작업 크기에 맞는 모드 선택
- 필요한 만큼만 읽고 바로 실행
- 병렬화는 read-only 또는 disjoint write scope에만 사용
- 변경 가까운 검증을 수행
- 최종 보고가 짧고 명확함

### 나쁜 orchestration 신호

- 작은 작업인데 deep 모드처럼 무거운 흐름 사용
- 조사만 길고 실행이 늦음
- 설계가 안 끝났는데 병렬 구현 시작
- 검증이 없거나 변경과 동떨어진 검증만 수행
- 결과 보고보다 절차 보고가 과도함

## 규칙 추출 기준

- 단일 run에서 보인 현상은 `관찰`로 남긴다
- 비슷한 패턴이 `3회 이상` 반복될 때만 강한 운영 규칙 후보로 승격한다
- 규칙 후보는 반드시 다음 형식으로 적는다

```md
- 규칙 후보: {짧은 문장}
- 근거: {반복된 패턴}
- 적용 조건: {언제}
- 기대 효과: {무엇이 줄거나 늘어나는지}
```

## 하지 말아야 할 것

- 측정하지 않은 시간을 숫자로 꾸며내지 말 것
- 고정 경로의 Claude 보고서가 있다고 가정하지 말 것
- 증거가 부족한데 강한 일반화를 하지 말 것
- 사용자가 요청하지 않은 memory, learnings, patterns 파일을 자동 생성하지 말 것
- 분석을 핑계로 실제 필요한 문서 수정이나 스킬 수정을 미루지 말 것
