# Orchestration

## 언제 사용하나

- 사용자가 `/orchestration`와 같은 의도로 복잡한 작업을 체계적으로 처리해 달라고 할 때
- 작업을 바로 시작할 수는 있지만, 먼저 범위, 의존성, 검증 순서를 정리해야 할 때
- 단순 실행보다 `분석 -> 수집 -> 계획 -> 실행 -> 검증`의 운영 품질이 중요한 작업일 때

## source mapping

- Claude command: `/orchestration`
- Source file: `claude/commands/orchestration.md`

## Codex 기준 핵심 해석

- 이 스킬은 Claude의 역할놀이형 멀티에이전트 체계를 그대로 재현하지 않는다.
- Codex에서는 `메인 에이전트 직접 수행`이 기본이고, 도구 병렬화와 계획 관리로 속도와 품질을 확보한다.
- 5단계 구조는 유지하되, 작업 크기에 따라 무게를 다르게 준다.
- 이 스킬과 예전 `subagent-convention` 문서가 충돌하면, `/orchestration` 실행 기준은 이 문서를 우선한다.

## 우선순위

1. 사용자의 실제 작업 완료
2. 최소 비용으로 충분한 맥락 확보
3. 충돌 없는 실행 순서 설계
4. 변경 범위에 맞는 검증
5. 짧고 선명한 결과 보고

## 실행 모드

`/orchestration`는 항상 같은 무게로 돌리지 않는다. 아래 기준으로 먼저 모드를 정한다.

| 모드 | 언제 쓰나 | 기대 동작 |
|------|-----------|----------|
| `quick` | 작은 수정, 좁은 범위 조사, 저위험 작업 | 읽기와 실행을 빠르게 압축하고 계획/보고는 짧게 |
| `standard` | 일반적인 기능 추가, 버그 수정, 소규모 리팩터링 | 기본 모드. 5단계를 유지하되 과잉 문서화는 하지 않음 |
| `deep` | 대형 리팩터링, 파일 이동, 다중 모듈 변경, 위험한 변경 | 의존성 확인, 범위 통제, 검증, 리스크 보고를 더 엄격하게 수행 |

### 모드 선택 규칙

`quick`은 아래를 모두 만족할 때만 쓴다.

- 예상 수정 범위가 대체로 `1~2개 파일`이다
- 파일 이동, 이름 변경, 스키마 변경이 없다
- 외부 조사나 장시간 계획이 필요 없다
- 검증도 `1개 명령` 또는 간단한 수동 확인으로 충분하다

`deep`은 아래 중 하나라도 맞으면 쓴다.

- 예상 수정 범위가 `8개 파일 초과`다
- 파일 이동, 이름 변경, import 경로 재배선이 있다
- 모듈 경계, API 계약, 데이터 구조를 건드린다
- 실패 비용이 높다
- 사용자가 깊은 분석, 구조 개편, 큰 정리를 직접 요청했다

그 외는 `standard`를 쓴다.

| 목적 | Codex 방식 |

## 기본 훅

현재 Codex 환경에 feedback hook이 있으면, `/orchestration`는 아래 훅을 기본 안전망으로 본다.

| 훅 | 기본 역할 |
|----|-----------|
| `SessionStart` | 세션 시작 기록 |
| `UserPromptSubmit` | 사용자 요청 기록 |
| `PreToolUse` | Bash 명령 시작 기록 |
| `PostToolUse` | Bash 명령 결과와 검증 분류 기록 |
| `Stop` | 고위험 turn의 실패 검증/검증 누락 점검 |

필수적으로 유용한 동작은 아래 정도다.

- 긴 `test`/`build`/`lint`/`typecheck` 완료 시 알림
- 고위험 turn에서 실패한 검증 직후 종료하려 하면 한 번 더 수리 유도
- 고위험 turn에서 저장소가 변경됐는데 검증 없이 종료하려 하면 한 번 더 검증 또는 사유 명시 유도

강제 원칙:

- 훅은 기본적으로 `advisory-first`여야 한다
- 작은 문서 작업까지 과하게 막지 않는다
- 사용자가 명시적으로 검증 불가 사유를 남기면 억지로 추가 검증을 강제하지 않는다

## 로그 정책

- 기본 로그 소스는 기존 local feedback hook 하나로 본다.
- `/orchestration`는 별도의 전용 타임라인, JSON report, 숨김 디렉터리 로그를 기본적으로 만들지 않는다.
- 즉, 기본값은 `별도 orchestration 로그 비활성`이다.
- 추가 로그나 report가 필요하면 아래 경우에만 opt-in으로 만든다.
  - 사용자가 명시적으로 요청함
  - `deep` 모드에서 장기 회고용 산출물이 실제로 필요함
  - 현재 hook 로그만으로는 필요한 증거가 부족함

중요:

- hook 로그와 별도 orchestration 로그를 둘 다 운영하는 구조를 기본값으로 두지 않는다
- 두 체계가 병행되면 중복, 드리프트, 회고 혼선이 생기기 쉽다

## 도구 선택 임계값

### `multi_tool_use.parallel`

아래 중 하나면 고려한다.

- 서로 독립적인 read-only 탐색이 `3개 이상`
- 서로 독립적인 검증 명령이 `2개 이상`
- 여러 파일을 읽어 비교해야 하지만 write는 아직 하지 않음

쓰지 말아야 하는 경우:

- 같은 파일이나 같은 write scope를 곧 수정할 예정
- 앞 결과가 다음 탐색 입력이 되는 순차 의존이 있음

### `update_plan`

아래 중 하나면 사용한다.

- 논리 단계가 `3개 이상`
- 작업이 한 번에 끝나지 않고 상태 추적이 필요함
- 사용자가 진행률을 보기 원하는 큰 작업임
- `deep` 모드로 분류됨

아래면 생략 가능하다.

- `quick` 모드
- 단일 수정과 단일 검증으로 끝나는 작업

### `spawn_agent`

기본값은 `사용 안 함`이다.

아래를 모두 만족할 때만 고려한다.

- 사용자가 하위 에이전트, 위임, 병렬 에이전트 사용을 명시적으로 원함
- 작업이 서로 독립적임
- write scope가 겹치지 않음
- 메인 에이전트가 그 결과를 기다리지 않고 다른 일을 진행할 수 있음

즉시 다음 단계가 그 결과에 막혀 있으면 메인 에이전트가 직접 처리한다.

## 기본 흐름 (5단계)

5단계는 유지하지만, 모드에 따라 압축하거나 확장한다.

| 단계 | 핵심 질문 | `quick` | `standard` | `deep` |
|------|-----------|---------|------------|--------|
| 1단계 요청 분석 | 무엇을 끝내야 하나 | 짧게 | 명시 | 성공 기준까지 정리 |
| 2단계 정보 수집 | 무엇을 읽어야 하나 | 최소 | 필요한 만큼 | 범위와 의존성까지 |
| 3단계 계획 수립 | 어떤 순서로 할까 | 머릿속 또는 짧게 | 간단한 작업 분할 | 명시적 계획과 추적 |
| 4단계 실행/검증 | 무엇을 바꾸고 어떻게 검증할까 | 빠르게 압축 | 기본 실행 | 충돌 통제와 검증 강화 |
| 5단계 결과 보고 | 무엇을 했고 무엇이 남나 | 짧게 | 표준 | 리스크와 회고 포함 |

### 1단계 - 요청 분석

- 사용자 의도, 완료 조건, 제약을 확인한다
- 관련 `AGENTS.md`, README, 현재 프로젝트 구조를 빠르게 본다
- 성공 기준과 중단 조건을 먼저 잡는다

### 2단계 - 정보 수집

- 관련 파일, 진입점, 기존 패턴, 테스트 경로, 실행 명령을 찾는다
- 목적은 충분한 맥락 확보이지, 전수조사가 아니다
- `deep` 모드에서는 import/참조 관계와 영향 범위를 더 엄격히 본다

### 3단계 - 계획 수립

- 수정 대상과 비수정 대상을 구분한다
- 작업 순서, 병렬 가능 여부, 검증 방법을 정한다
- 파일 이동, rename, 큰 리팩터링이면 참조 경로를 먼저 확인한다

### 4단계 - 실행 및 검증

- 수정 전에 무엇을 바꾸는지 짧게 알린다
- 구현은 가능한 한 바로 진행한다
- 검증은 아래 `검증 사다리`에서 가장 싼 단계부터 시작하되, 위험도에 따라 올린다
- 같은 파일이나 강하게 결합된 모듈은 순차 처리한다

### 5단계 - 결과 보고

- 무엇을 바꿨는지
- 무엇으로 검증했는지
- 무엇이 남았는지
- `deep` 모드이거나 재작업이 컸으면 짧은 회고까지 남긴다

## write scope 규칙

병렬화 판단은 `작업 수`보다 `write scope`를 우선한다.

병렬화 가능한 경우:

- read-only 조사
- 서로 다른 디렉터리의 독립 수정
- 독립 테스트/로그 수집

순차가 기본인 경우:

- 같은 파일 수정
- 같은 모듈의 상호 의존 로직 수정
- 설계 결론이 나와야 다음 변경이 가능한 경우

애매하면 순차로 간다.

## 검증 사다리

가장 싼 검증부터 시작하되, 변경 위험이 높으면 한 단계 이상 올린다.

| 변경 유형 | 기본 검증 | 필요 시 상향 |
|-----------|----------|--------------|
| 문서/프롬프트/스킬 문서 | diff 검토, 렌더/문법 확인 | 링크/참조 검색 |
| 설정 파일 | 문법 확인, 관련 명령 dry-run | 린트, 앱 시작 확인 |
| 라이브러리/백엔드 코드 | 관련 테스트 또는 정적 확인 | 타입체크, 린트, 빌드 |
| 프론트엔드 코드 | 타입체크, 린트, 관련 테스트 | 빌드, 수동 UI 확인 |
| 파일 이동/이름 변경 | 참조 검색, import 경로 확인 | 테스트, 빌드 |
| 배포/인프라 스크립트 | dry-run 또는 정적 점검 | 샘플 실행, 관련 명령 검증 |

검증 원칙:

- 의미 없는 전체 테스트보다 변경과 가까운 검증을 우선한다
- repo에 lint 명령이 있으면 `lint`는 기본 검증 후보에 포함한다
- 검증을 못 했으면 이유를 남긴다
- 실패한 검증은 숨기지 않는다

## 중단 및 에스컬레이션 조건

아래 상황이면 바로 밀어붙이지 말고 짧게 정리해서 사용자에게 물어보거나, 최소한 결과에 강하게 표시한다.

- 같은 파일에 사용자 변경과 내 변경이 직접 충돌함
- 파괴적 명령이나 데이터 손실 가능성이 있음
- 필수 환경변수, 접근권한, 외부 리소스가 없어 핵심 작업을 진행할 수 없음
- 요구사항 해석이 둘 이상 가능하고 결과 차이가 큼
- 검증 실패 원인이 여러 갈래라 안전한 자동 수정이 어려움

## 보고 형식

원본 Claude 문서의 `타임스탬프 로그 + JSON 저장`은 Codex에서 기본 의무가 아니다.

Codex에서는 아래 네 줄기가 기본이다.

- 결과
- 검증
- 남은 리스크
- 다음 액션이 필요하면 그것만 짧게

`deep` 모드에서만 선택적으로 회고를 덧붙인다.

- 병목이 어디였는지
- 병렬화가 실제로 이득이 있었는지
- 다음 번에는 무엇을 줄일 수 있는지

## 실행 예시

```text
/orchestration 사용자 인증 기능을 추가하세요

모드 판정: standard

1. 요청 분석
   - 인증 방식, 기존 미들웨어, 성공 기준 확인

2. 정보 수집
   - 관련 라우트, 사용자 모델, 미들웨어, 테스트 파일 탐색
   - 독립적인 파일 읽기는 병렬 처리

3. 계획 수립
   - 모델 -> 서비스 -> API -> 검증 순서 정리

4. 실행 및 검증
   - 코드 수정
   - 관련 테스트와 타입체크 실행

5. 결과 보고
   - 변경 내용, 검증 결과, 남은 리스크 요약
```

## 하지 말아야 할 것

- Claude 전용 모델명, `Task(...)`, 역할놀이식 위임 체계를 실제 실행 방식처럼 적지 말 것
- 작은 작업도 무조건 무거운 5단계 보고서로 처리하지 말 것
- 사용자가 요청하지 않은 JSON 보고서나 숨김 디렉터리 로그 파일을 만들지 말 것
- 병렬화 가능성만 보고 같은 write scope 작업을 동시에 던지지 말 것
- 조사만 길게 하고 구현을 뒤로 미루지 말 것
- 하위 에이전트가 허용되지 않았는데 이를 기본 전략처럼 쓰지 말 것
