# Subagent Convention

Codex에서 하위 에이전트를 쓸 때 따르는 기본 규칙이다.

## 기본 원칙

- 기본값은 `메인 에이전트 직접 수행`이다.
- 하위 에이전트는 항상 이득이 있을 때만 쓴다.
- 사용자가 위임, 병렬 에이전트, sub-agent 사용을 명시적으로 원할 때만 `spawn_agent`를 고려한다.
- 단순 read-only 탐색은 하위 에이전트보다 `multi_tool_use.parallel`이 먼저다.
- 즉시 다음 단계가 그 결과에 막혀 있으면 메인 에이전트가 직접 처리한다.

## 역할

| 역할 | 언제 쓰나 | 책임 |
|------|-----------|------|
| 메인 에이전트 | 항상 기본 | 계획, 사용자 소통, 통합, 최종 검증 |
| `explorer` | 코드베이스 질문, read-only 조사 | 좁은 질문에 대한 사실 수집과 답변 |
| `worker` | 분리 가능한 구현/수정 작업 | 지정된 파일 범위 안에서 구현과 검증 |
| `default` | explorer/worker로 딱 잘라 분류하기 어려움 | 일반 보조 작업 |

## 언제 위임하나

- 독립적인 코드베이스 질문이 `2개 이상`이고 메인 에이전트가 동시에 다른 구현을 진행할 수 있을 때
- 서로 다른 write scope를 가진 구현 작업을 분리할 수 있을 때
- 장시간 검증이나 로그 수집을 옆에서 돌려두는 것이 메인 흐름을 막지 않을 때

## 언제 위임하지 않나

- 작은 작업이거나 한 번 읽고 바로 수정 가능한 경우
- 같은 파일, 같은 모듈, 같은 write scope를 여러 작업이 건드리는 경우
- 다음 행동이 그 결과에 즉시 의존하는 경우
- 위임 프롬프트를 쓰고 기다리는 비용이 직접 처리보다 큰 경우

## write scope 규칙

- 하나의 파일 또는 강하게 결합된 모듈은 한 에이전트가 책임진다.
- worker는 자신에게 할당되지 않은 파일을 넓게 건드리지 않는다.
- worker는 다른 사람이나 다른 에이전트의 변경을 되돌리지 않는다.
- worker 프롬프트에는 반드시 담당 파일 또는 모듈 범위를 적는다.
- worker 결과에는 변경 파일 목록과 검증 결과가 포함되어야 한다.

## 권장 흐름

1. 메인 에이전트가 먼저 전체 목표와 즉시 해야 할 로컬 작업을 정한다.
2. 병렬로 돌릴 수 있는 sidecar 작업만 분리한다.
3. read-only면 `explorer`, 구현이면 `worker`를 고른다.
4. 범위, 파일, 출력 형식을 짧고 명확하게 적어 보낸다.
5. 기다리는 동안 메인 에이전트는 겹치지 않는 로컬 작업을 진행한다.
6. 필요할 때만 `wait_agent`로 결과를 받고 통합한다.

## 프롬프트 작성 규칙

### explorer용

- 질문은 하나의 초점으로 좁힌다.
- 기대 출력은 `관련 파일`, `핵심 패턴`, `주의점` 정도로 제한한다.
- 읽기 전용이라는 점을 분명히 한다.

예시 구조:

```text
목표: {알아내고 싶은 것}
범위: {디렉터리/모듈}
질문:
- {구체 질문 1}
- {구체 질문 2}
출력:
- 관련 파일 경로
- 현재 패턴 요약
- 주의할 의존성
수정 금지
```

### worker용

- 담당 범위를 명확히 적는다.
- 다른 에이전트도 동시에 작업 중일 수 있다고 알려준다.
- 수정 목표, 허용 범위, 검증 명령을 함께 적는다.

예시 구조:

```text
담당 범위: {파일/모듈}
목표: {구현 또는 수정 목표}
제약:
- 당신은 코드베이스에 혼자가 아니다
- 다른 사람의 변경을 되돌리지 말 것
- 담당 범위를 벗어난 수정은 최소화할 것
검증:
- {명령 또는 확인 방법}
최종 보고:
- 변경 파일
- 핵심 변경
- 검증 결과
```

## wait 규칙

- 결과가 필요할 때만 기다린다.
- 습관적으로 polling하지 않는다.
- 이미 충분한 로컬 작업이 있으면 먼저 그 일을 한다.

## 통합 체크리스트

- 변경 파일 범위가 원래 위임 범위와 맞는가
- 메인 작업과 충돌이 없는가
- 검증이 실제로 수행됐는가
- 추가 수정이 필요하면 메인 에이전트가 마무리할지, 다시 좁혀서 위임할지 결정했는가

## 하지 말아야 할 것

- 하위 에이전트를 기본 전략처럼 쓰지 말 것
- 같은 파일 수정 작업을 여러 worker에 나눠 던지지 말 것
- 메인 흐름의 즉시 blocker를 무조건 위임하지 말 것
- `codex exec`를 중첩 호출하는 식으로 Codex 안에서 다시 Codex를 띄우지 말 것
